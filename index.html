The Hub Trilogy

Team intelligence system — spreadsheet to 100-person enterprise production deployment across three builds

Expand
PythonFastAPIPydanticSQLiteDockerD3.jsSupabasePostgreSQLEnterprise PaaSEnterprise SSOEnterprise CI/CDRBACDSID Identity

Three builds. One through-line. The spreadsheet was the schema. The public site was the interface lab. The Supabase build validated the backend. The Shuri deployment was the capstone. Each build had a specific job. The three versions aren't iterations of the same project — they're the process: figure out how to build it, prove the backend works, then ship the real thing.

Background

When the a global compliance team expanded from 13 people to 100 globally, the coordination infrastructure didn't scale. There was no central view of what projects were active, who was working on them, or what the team's collective skills looked like. Leads couldn't answer basic questions: who has capacity right now? Who knows how to do X? What are we actually working on? The starting point was a spreadsheet — a form sent to the full team asking for their skills, expertise, hobbies, and current work. Those responses became the bones of the data model. The data existed before the system did.

Stage 1 — Team Intelligence Hubthevillanelle.github.io/team-intelligence-hub

The learning build. Single HTML file, data embedded as JavaScript objects, deployed via GitHub Pages. The goal wasn't to ship a product — it was to answer: what does this actually need to do, and how do I build it? Core UI patterns worked out here: project cards, profile cards with traffic-light availability indicators (green/yellow/red based on project count), skills search, D3.js force graph for team connections. PIN-gated admin layer stood in for real authentication. All data hardcoded and static. Getting the interface right in the cheapest possible environment meant that by the time a real backend was needed, the product logic was already settled.

Stage 2 — m'atelier Studiothevillanelle.github.io/studio

The backend proof of concept. Domain shifted to personal project management, but the architecture question was the same: can this system persist real data, sync across devices, handle writes without a server? Supabase answered: PostgreSQL + REST API + Row Level Security, zero infrastructure to manage. Schema designed from scratch: team_members (work_skills[], non_work[], hobbies[], private_note), projects (task_1/2/3, volunteer_need, contact_id FK), project_assignments (role_label), knowledge_nodes (category, weight 1–5 foundational scale), member_knowledge junction. The fetch calls in the frontend mapped almost 1:1 to what the FastAPI layer would later do in production. This version also introduced the knowledge mind map: a D3 force graph spanning Web Dev, Data Viz, Infrastructure, AI Tools, Design, Strategy, Security, Finance, and Blockchain. Labels appear progressively as you zoom in — category tags at scale > 0.5, full labels at > 1.1, description snippets at > 3. Three separate SVG text layers managed independently.

Stage 3 — Skills Hub (Production)thevillanelle.github.io/team-intelligence-hub

The full version: real authentication, real persistence, real deployment, real users. System architecture: Browser → HTTPS → the enterprise PaaS, which injects x-user-email + x-user-id (DSID) headers → FastAPI (Python 3.12, Uvicorn ASGI) → SQLite (WAL mode, seeded from employee JSON on first boot). Why FastAPI: Pydantic models gave automatic request validation and typed contracts. Async lifespan hooks handled DB initialization cleanly. Self-documenting API at no extra cost. Why SQLite with WAL mode: at 100 users with read-heavy patterns, a managed database would have been overengineered. Write-Ahead Logging handled concurrency safely. Choosing not to use a managed database was the right call. Identity resolution: The PaaS injects x-user-email and x-user-id headers on every authenticated request — confirmed by building a debug endpoint and dumping live request headers in production. Final implementation checks DSID first (immutable), falls back to email, falls back to DEV_USER_EMAIL for local dev. Deployment: ci.yml defines the CI/CD pipeline — vessel build, Memento signing, push to enterprise container registry, deploy to platform via deploy.yaml. Fires on every push to main. Two-region (us-west-2 + us-east-1) with traffic-based autoscaling. Dockerfile uses multi-stage build with non-root appuser — a Shuri requirement and container security best practice.

The Standing System

The availability traffic light answers one question: what is this person's current bandwidth? Standing answers a different question: has this person been reviewed by their lead recently? Every 90 days, all standing values automatically flip to red. On sign-in, team leads see an alert for their direct reports and must confirm each one, resetting to green with a timestamp. Leads can manually flag someone red at any time with a required written note. The 90-day trigger is a forcing function, not a reminder. Confirmation is required, not optional. The timestamp is stored — the review history is auditable. An earlier version ran two separate cycles (90-day eligibility reset and 60-day standing review). Identifying that redundancy and collapsing it into one coherent system — one mechanism, one cycle, two surfaces — was a design improvement shipped during active development.

Outcome

Production system live on enterprise internal infrastructure. Two-region deployment. 100 users. Full audit and standing confirmation system replacing manual spreadsheet processes. Public portfolio version (thevillanelle.github.io/team-intelligence-hub) accessible from any network with zero external dependencies. Supabase-backed personal version (thevillanelle.github.io/studio) validating the PostgreSQL approach that informed the production design. End-to-end solo: concept, data collection, design, development, and deployment.

Risk Training Academy

Full training platform — knowledge base, decision matrix, scored simulator, and geography game in a single HTML file

Expand
HTMLCSSJavaScriptFuse.jsD3.jsCustom ScoringFisher-YatesCSS Custom PropertieslocalStorageIntersectionObserverGitHub PagesGitHub Actions

The team had no searchable knowledge base, no interactive decision tool, and no training simulator. This built all four — plus a standalone geography training game — with zero backend, zero build step, and zero dependencies. The entire platform is a single index.html — 2,700+ lines of pure HTML, CSS, and JavaScript. Deployed to the internal network. Works on mobile.

Background

The investigations team works a high-stakes queue: sanctions screening across multiple clients (AMP, WWDR, IdMS), four case types, dozens of edge cases, and procedure documents scattered across an internal site with broken deep links and a Cross-Origin-Opener-Policy header that blocks direct external navigation. New investigators had to learn the system by reading dense procedure docs in isolation, with no way to practice before hitting a live queue. The tools existed. The training infrastructure didn't. The gap between "read the procedure" and "work a case confidently" was the problem to solve.

The Codex — Knowledge Base

A fully searchable reference system covering all four compliance procedure documents, a 40+ term glossary, and reference guides — organized by client, document type (Original summaries vs. Tech Edits), and category. The search implementation is hand-rolled: a scoring function that weights title matches highest, then tag matches, then body frequency (capped per word to prevent long documents from dominating). Results render live on input with a 180ms debounce and an animated searching indicator. Highlight injection marks matched terms in results. The full KB array — 1,000+ lines of compliance content — is embedded directly in the file as a JavaScript array, making the tool work on any device on VPN with no API calls. The COOP header workaround: every relevant card includes a "📋 Copy source URL" button that copies the deep link to clipboard — identified, documented, explained in the User Guide, and routed around without any server-side changes. A rotating slideshow cycles through 25 key compliance terms as search prompts — clickable, auto-advancing at 3.5-second intervals, with dot navigation and keyboard support.

The Oracle — Decision Matrix

An interactive step-by-step decision tree for all four case types: With PII, IdMS Without PII, Companies, and Batch Free Developer. Investigators answer branching questions — entity type, name match, DOB, location — and receive the correct action (Clear, Escalate, or Suspend), the exact reason code, a plain-English explanation, and a full breadcrumb trail of their decision path. A progress bar tracks position through each tree. Built from the official decisioning matrix reference document (CR0007) plus procedural context from four client procedure docs. The matrix logic was written to match the source documents exactly — including edge cases like China company naming, the full street address override, and the "previously suspended" batch_free_developer scenario. The Decision Matrix trees are plain JavaScript objects. Traversal is a simple recursive function. Adding a new matrix type requires one new object in TREES.

The Trials — Training Simulator

40 scenarios across three difficulty levels. Each scenario presents a realistic compliance case: screened entity name, location, PII status, matched data hits, client and event type. Investigators choose the correct action from four options and receive immediate feedback. Wrong answers are treated as the most educational outcome — every incorrect answer shows the full correct reasoning before advancing. The game tracks score, lives (3 per session), rank progression (Trainee through Lead Investigator), case count, and streak (+50 bonus at 3+ consecutive). Scenarios shuffle on each session start using Fisher-Yates, ensuring no two sessions have the same case order. Difficulty design: Initiate (13 scenarios) — clear variances, single-factor decisions, entity type mismatches, false positives. Investigator (13 scenarios) — China naming conventions, 2/3 name match with MZ account age, expired PII, AKA traps, YOB shortcuts. Olympian (14 scenarios) — full address match exception, ownership percentages, previously suspended companies, the deceased-date-only escalation, system notes traps, the gibberish flag, the Iran connection override.

Where in the World? — Geography Training Game

A standalone geography and language identification game embedded in the platform. Four modes, 135 questions: Script Intel (identify writing systems from examples, including compliance-critical cases like 조선 and ایران), Country Decoder (identify countries from native-language names, including all three sanctioned countries), Name Origins (identify geographic origin: Zhang Wei, José, Reza, Nguyễn, Giuseppe), Sanctioned or Safe? (39 questions covering all three sanctioned countries, all five Ukrainian sanctioned regions, and real city-level address scenarios). Compliance-critical questions flagged red, worth 150 points. An investigator who can't recognize that ایران is "Iran" in Persian script is more likely to miss a true match. Geography training is compliance training.

Location Themes + Tools Page

Nine distinct visual themes — one per office where the team operates: Austin, Elk Grove, San Diego, Arizona, Raleigh, Boston, New York, Singapore, and a default Global theme. Each theme remaps the full CSS custom property set with location-specific visual signatures: brick patterns for Boston, a neon grid for Singapore, a lone-star watermark for Austin, a sunset radial for San Diego. Theme and dark/light toggle persist in localStorage. The themes weren't cosmetic — investigators in Singapore and Raleigh see the same product built for them specifically. The Tools page houses corrected date conversion utilities for Japanese imperial era dates and Taiwanese ROC calendar dates. The Japanese converter corrects an identified error in the official Job Aids page: the Shōwa era formula shown on the Job Aids site is off by one year. The correction (Shōwa Year 1 = 1926, not 1927) is documented, sourced (JACAR archives), and flagged with a visible warning banner. The error was identified by an investigator in production; the correction was shipped here while the official Job Aids team was notified.

Outcome

Full training and reference platform live on the internal network, accessible via VPN from any device. 40 compliance training scenarios across 3 difficulty levels. 135-question geography training game with compliance-critical flagging. Interactive decision matrix for all 4 case types. Searchable knowledge base with 40+ glossary terms and full procedure document summaries. Corrected date conversion tools with documented bug fix. 9 location-specific visual themes covering every office. Zero external dependencies, zero backend, zero build infrastructure — deployed as a single file. Full platform: thevillanelle.github.io/risk-training-academy

EQX Doubles Finder

Workout scheduling across 36 NYC Equinox locations — three complete platform rewrites, one PostgreSQL function throughout

Expand
PostgreSQLPLpgSQLSupabaseJavaScriptPythonFlaskJinja2SwiftSwiftUIXcodeGitHub ActionsRailwayMVVMAsync/AwaitHaversineMTA Data

A personal problem built to production across three complete platform rewrites. The database layer was designed once and survived all three. Every architectural decision was driven by a concrete limitation encountered in production. The project wasn't planned in three phases — it was iterated into them.

Background

Booking back-to-back Equinox classes across New York's network is genuinely complex: which two classes pair well by type, do the times actually work, is the second location walkable or a 40-minute subway ride, which clubs have a pool or steam room. The network spans 36 locations across Manhattan and Brooklyn. No tool existed to answer these questions together. The mental overhead was real and recurring, so the tool got built.

Phase 1 — The SPA

Browser → Supabase direct. GitHub Actions ingestion script pulls class schedule PDFs every 6 hours, loading data into PostgreSQL via Supabase. The core pairing logic lives in find_doubles(), a PostgreSQL stored procedure written in PLpgSQL, invoked via db.rpc(). It takes eight parameters: array of selected club IDs, day of week (as a string like MONDAY), two category values, max gap in minutes, pair order constraint (either, 1first, or 2first), and a time window expressed as minutes-since-midnight for both start and end bounds. The class category system is the most opinionated part: each category carries a kw keyword array that find_doubles() uses to match against class names. Seven groups, 23 specific options. The Sculpt and Strength group alone resolves against 30+ class name variants — from metcon and tabata to forza, vipr sculpt, and firestarter. Cycling resolves beats ride, rhythm ride, anthem ride, pursuit, precision ride, endurance ride. Boxing resolves rounds: boxing, rounds: kickboxing, rounds: bags, cardio kickboxing, knockout, powerstrike, shadow-do, muay thai, and zen combat. Transit router: buildTransitCache() pre-computes the transit relationship between every pair of clubs — 36 × 35 / 2 = 630 pairs — on boot. Four transit types: same, walk (haversine distance, 1.3 route factor, 80m/min pace), quick-train (shared subway line, under 4km), extended (transfer required or over 4km). Each type carries a minGap. The TIGHT flag fires when a pair's gap_minutes is less than the transit's minGap. Transit routes render with MTA official color assignments: A/C/E blue, B/D/F/M orange, 1/2/3 red, 4/5/6 green, 7 purple, N/Q/R/W yellow. Boot behavior: two things run in parallel — splash screen minimum 2,800ms and loadTomorrowPreview() fires a full RPC scan of tomorrow's schedule. pickTop3() uses geographic profiling: morning pairs matched to downtown/Brooklyn clubs, afternoon to midtown, evening to uptown. Result: a curated landing screen that reflects real NYC workout geography. Problem: credentials visible in HTML source, slow load times.

Phase 2 — The Python Rewrite

Browser → Flask → Supabase. Python is better for scraping, so the rewrite moved to Flask. Flask receives POST requests at /api/doubles, calls Supabase via the Python SDK, returns JSON. Credentials moved off the client. The CLUBS list and CATEGORIES data that were JavaScript objects are now Python lists of dicts in app.py, passed into Jinja2 templates at request time. Club location checkboxes and class type dropdowns generated with {% for club in clubs %} loops. Interactive behavior — fetching from /api/doubles, rendering pair cards, transit routing — still JavaScript in the browser, carried over from Phase 1. Project structure: app.py holds all routes, data definitions, and app entry; supabase_client.py is a one-file module reading credentials from environment variables; requirements.txt lists three dependencies (flask, supabase, python-dotenv); Procfile and railway.json handle Railway deployment. find_doubles() unchanged — same function, same parameters, same results — now called from Python via the Supabase Python SDK.

Phase 3 — The iOS App

Swift + SwiftUI + Xcode. Same Supabase backend. Same find_doubles() function via Supabase's REST API. What changed is everything about how the code executes. Project layered cleanly: Models/ (Club, DoubleResult, SearchParams as typed Swift structs with Codable for automatic JSON decoding), Services/ (SupabaseService.swift, all HTTP via async/await), Views/ (SearchView, ResultsView, ResultCard, ClubsView, SplashView as discrete SwiftUI components), ViewModel (SearchViewModel, @Published properties, reactive UI). TransitHelper.swift ports the haversine distance calculation and subway line routing from the JavaScript version into Swift, surfacing the same transit type classifications and nearest subway lines per club location.

The Through-Line

find_doubles() was written once, in PLpgSQL, in Phase 1. It ran in Phase 2 behind a Flask server. It runs in Phase 3 behind a SwiftUI app. Three complete rewrites of the presentation and infrastructure layer, zero rewrites of the core logic. The algorithm lives where the data lives. The application layer is replaceable. Working app across three platforms. 36 NYC locations mapped with lat/lng, neighborhood groupings, and subway line data. 23-category class taxonomy with keyword arrays resolving 100+ class name variants. Client-side transit router pre-computing 630 club-pair relationships on boot with MTA subway line badges. Automatic tomorrow preview using geographic profiling.

The Engagement Architecture

Four compounding builds across one rotation — from a live event to a planning system to a summit proposal

A rotation onto the Employee Engagement and Communications team created both the access and the time to actually do something about a set of visible gaps. No gap was just identified and named — each one was answered with something built.

Stage 1 — EEC Rotation Training Site

The engagement team's onboarding material was a document. Fourteen tools to learn, spread across wikis and shared folders, with no priority ordering, no progress tracking, no single landing place. The first build turned it into an SPA: 14 tool cards organized by priority tier (High/Medium/Low/Additional), a sticky live progress bar, staggered card entrance animations, and a visual onboarding timeline showing which tools to hit in weeks 1, 2, and 3. Checkboxes persist via localStorage — progress survives a page refresh. Password reveal chips for internal tool credentials. Light/dark toggle. The design choice was deliberate: a progress bar at 0% is motivating in a way that a bullet list is not.

Stage 2 — AHA After School Hangout

A fully planned and successfully executed virtual team engagement event — a retro arcade aesthetic (CRT scanline overlay, neon glow, Press Start 2P pixel font, animated neon-pulse hero title, full light/dark/accessibility mode trifecta). Five game categories: Arcade Classics (Atari Breakout, Minesweeper, Bejeweled — high score competition), Creative and Social (Codenames, Gartic Phone), Brain Teasers (Skribbl.io, Openguessr, Kahoot!), Variety Packs (Pogo, Google Arts and Culture), and an After School Special watch section (Magic School Bus, Totally Spies, Looney Tunes). Three prize categories — Activity Passport Champion, High Score Heroes, and Raffle — with tooltip hover reveals. The submission flow validates @company.com email, opens a pre-filled mailto: with the entry structured as a formatted email body. On submit, the form replaces itself with a success state. The event ran. It was a success. Public event hub: thevillanelle.github.io/gather

Stage 3 — Engagement Hub

The event proved the model. It also surfaced the planning problem: no system for figuring out what kind of event to run, who was available when, or what the team actually wanted. Four modules: (1) Event Planning Portal — a 10-step questionnaire that scores 12 event blueprints against input, weighted by category match, format, budget, headcount, feeling goal, energy, and context. Results render as ranked recommendation cards with expandable logistics checklists and Webex integration tips. (2) AI Agent Setup Guide — step-by-step guide for building a custom Enchanté agent for EEC event planning with a copyable starter system prompt, token explainer, and 6-entry FAQ accordion. (3) Join the Team — 10-step employee intake survey collecting name, team, city, work arrangement, birthday/Appleversary, energizers (multi-select chips), time preference, event types, group size preference, dietary needs, and open notes. Profile saves to localStorage and feeds Module 4. (4) What We Love — A force-directed canvas interest graph built without D3, custom physics engine in raw JavaScript. 22 interest nodes (Time Preference, Energizers, Event Types, Group Size) orbit in a ring. Member dots connect to their linked interest nodes via spring physics. Drag-to-pan, scroll-to-zoom, hover tooltips with member name, team, location, and interest cluster. Category filter buttons. Live search by name. Insight bar showing top interests ranked by frequency. The graph isn't decorative — it answers the question event planners need answered before they plan anything.

Stage 4 — The Proximity Summitthevillanelle.github.io

~150 employees working from home across 40+ cities — no dedicated programming, limited leadership access, no shared experience that makes a dispersed community feel like one. The Proximity Summit is a fully designed, fully documented proposal for a 4-day annual virtual summit. The proposal site is a full SPA with fixed nav, Leaflet.js map, tabbed agenda, accordion FAQ, and a master 20-week project timeline. Dark/light toggle with CartoDB tile layer swap on the map. The map reveal: 150 dots, one per employee, distributed across 40+ cities — no words needed for the first few seconds, just scale. Dots rendered as styled divIcons with glow shadows, population-weighted using 50 real US city coordinates with per-city jitter, geographic bounds checking to prevent dots from landing in Mexico. Four days with distinct identities: Day 1 (Open) — map reveal, icebreaker, growth panels, Coffee With space open; Day 2 (Showcase) — AHA Showcase goes live, peer submissions featured; Day 3 (Grow) — deepest panels, leader 1:1 slots; Day 4 (Close) — AI Wrapped experience, Virtual Arcade offsite. Hourly employee protection framework: session tagging system (🟢 Drop In, 🟡 Schedule Required, 🔵 Sign Up, 🟣 Leader Present), manager pre-clearance, pre-allocated time budgets. Six leader participation tiers. 17-question leader FAQ written from the leader's perspective, answered honestly. Budget: $15K total versus $100K+ for an in-person equivalent.

Interactive onboarding checklist with live progress tracking, replacing a static document. Successful virtual event planned, executed, and hosted end-to-end. Seed Your Ideas: Compliance

End-to-end idea submission program with a full automation layer — built entirely on tools the team already had open

Expand
Slack WorkflowsQuip AutomationMCP IntegrationAI Agent DevEnchantéProgram DesignSystems ArchitectureTechnical Writing

Phase 1 built the intake pipeline. Phase 2 closed a feedback loop that Phase 1 didn't solve — identified through direct program evaluation and fixed with a second structural build. Every piece of the system, every document, and every communication asset was designed and produced by one person.

Background

SDS had an investigator idea submission program. Compliance didn't. As the team expanded, there was no structured way for investigators to surface ideas, no place for leadership to track them, and no visibility into what was being considered or actioned. The gap was identified and the case for a compliance-specific program was made.

Phase 1 — Building the Pipeline

The design constraint was adoption: a system that required new tools or new behaviors wouldn't get used. The pipeline was built entirely on Slack and Quip — tools the team already had open every day. A Slack workflow was designed and deployed as the submission interface. Investigators fill out a structured form — name, category, idea description, business impact — without leaving Slack. The workflow populates a purpose-built Quip tracking spreadsheet automatically, giving leadership a centralized, real-time view of all submissions. Built in partnership with Sarah. Handed off to Julianne as program leader, responsible for day-to-day management, idea evaluation, and working submissions with the broader leadership team.

The Gap Identified

Quarterly presentations were established as the communication mechanism after handoff. Through those presentations, a structural problem became clear: investigators were submitting ideas and hearing nothing back. No acknowledgment, no status update, no close-out. They had no way to know whether their idea had been seen, considered, actioned, or shelved. The quarterly format was reporting to leadership, not communicating back to investigators. Something in the architecture needed to change.

Phase 2 — Closing the Loop

A new system was designed with one binding constraint: it couldn't add workload to the leadership team already running the program. All trigger points had to map to things leaders were already doing in Quip. Quip structural redesign: clearly designated internal notes column (excluded from all automated outputs), Actions Taken column as single source of truth for close-out communications, format guidelines defined to ensure clean AI agent output. Quip-to-Slack integration: edits to the document — including the Complete checkbox trigger — post automatically to the channel. A leader notification step was also added to the original Slack workflow, triggering an alert on every new submission. Four email templates, designed and copy-edited to organizational voice standards, mapped to the four states an idea can be in: Stage 1 (submission confirmation), Stage 2 (active review notification, triggered when the first Actions Taken entry is logged), Stage 3A (implemented — full close-out with a dated timeline compiled from the Actions Taken history), Stage 3B (tabled/declined — honest, respectful close-out with clear reasoning and an actionable next step). Every trigger point maps to an action a leader was already taking. No new behaviors required.

The Custom AI Agent

A custom agent was built in Enchanté and connected to the Quip document via MCP (Model Context Protocol) integration. The agent reads live data directly from the spreadsheet, compiles the Actions Taken history, and produces a formatted, policy-compliant email draft on demand — in the correct organizational voice, with team member names never surfaced to investigators. Any team member can generate the correct email for any stage with a single sentence prompt. The agent's instructions, constraints, tone standards, privacy rules, and operational logic were written from scratch. The goal was a tool any member of the team could pick up and use correctly without knowing how it was built.

Program Asset Suite

Leader program overview (visual explainer of the full pipeline, PDF-ready and email-safe). Investigator guide (plain-language visual guide including what communications to expect at each stage and when). Full email journey reference (all four templates in sequence with trigger annotations). Seed Your Ideas: Compliance setup guide (step-by-step Enchanté onboarding with agent instructions ready to copy and deploy). General Enchanté getting started guide (broader onboarding covering GenAI training, Homebrew, L'Atelier, MCP servers, tokens, and custom agent creation). Leadership elevator pitch. Investigator Slack blast.

Outcome

43+ ideas collected across multiple submission cycles. Over half actioned or completed, with documented outcomes for each. 4 email templates designed, copy-edited, and ready to deploy against the existing backlog. 1 custom AI agent live and operational — close-out email drafting reduced to a single prompt. 6 program documents covering leadership, investigators, and team onboarding. Zero additional tooling required — the full system runs on Slack, Quip, and Enchanté. Zero ongoing maintenance burden — the agent handles drafting, Quip-to-Slack handles alerting. Any team member can run the program end-to-end using the shared documentation and agent.

About

As a Developer

The pattern across every project here: work out what the thing needs to do before committing to how it's built. The public site is the interface lab — no backend, no infrastructure, just figuring out the product logic. The proof of concept validates the backend approach. The production deployment is the capstone. That sequence isn't a methodology — it's what happens when each build is taken seriously enough to expose its own limitations. The interface stays stable because it was designed before it was coupled to anything.

As a Program Designer

Every program in here was designed to run after its builder hands it off. The AI agent handles drafting on demand. The Quip-to-Slack integration handles alerting automatically. The documentation lets any team member pick it up without a briefing. Designing for your own absence is harder than designing for your own presence — it requires knowing exactly what decisions need to be made, and building the system so those decisions are obvious to anyone who comes after you.

As a Builder

Browser SPA. Python server. Native iOS. Apple enterprise infrastructure with a two-region deployment. Slack and Quip automation layer with a custom AI agent. No two projects solved the same problem or used the same stack. Each one followed its natural next question — what does this actually need, and what happens when you take that seriously rather than stopping at something that already works? That's the range. It wasn't planned. It accumulated.
