The Grid

Team intelligence system — from a survey spreadsheet to a three-stage production deployment across public, Supabase, and enterprise infrastructure, rebranded and extended in 2026

Expand
PythonFastAPIPydanticSQLiteDockerD3.jsSupabasePostgreSQLEnterprise PaaSEnterprise SSOEnterprise CI/CDRBACUser Identity TokenpytestDatadog

Three builds. One through-line. The spreadsheet was the schema. The public site was the interface lab. The Supabase build validated the backend. The enterprise PaaS deployment was the capstone. Each build had a specific job. The three versions aren't iterations of the same project — they're the process: figure out how to build it, prove the backend works, then ship the real thing. Rebranded and extended in 2026 with a full test suite and production observability layer.

Background

When the organization's Compliance team expanded from 13 people to 100 globally, the coordination infrastructure didn't scale. There was no central view of what projects were active, who was working on them, or what the team's collective skills looked like. Leads couldn't answer basic questions: who has capacity right now? Who knows how to do X? What are we actually working on? The starting point was a spreadsheet — a form sent to the full team asking for their skills, expertise, hobbies, and current work. Those responses became the bones of the data model. The data existed before the system did.

Stage 1 — Team Intelligence Hubthevillanelle.github.io/team-intelligence-hub

The learning build. Single HTML file, data embedded as JavaScript objects, deployed via GitHub Pages. The goal wasn't to ship a product — it was to answer: what does this actually need to do, and how do I build it? Core UI patterns worked out here: project cards, profile cards with traffic-light availability indicators (green/yellow/red based on project count), skills search, D3.js force graph for team connections. PIN-gated admin layer stood in for real authentication. All data hardcoded and static. Getting the interface right in the cheapest possible environment meant that by the time a real backend was needed, the product logic was already settled.

Stage 2 — m'atelier Studiothevillanelle.github.io/studio

The backend proof of concept. Domain shifted to personal project management, but the architecture question was the same: can this system persist real data, sync across devices, handle writes without a server? Supabase answered: PostgreSQL + REST API + Row Level Security, zero infrastructure to manage. Schema designed from scratch: team_members (work_skills[], non_work[], hobbies[], private_note), projects (task_1/2/3, volunteer_need, contact_id FK), project_assignments (role_label), knowledge_nodes (category, weight 1–5 foundational scale), member_knowledge junction. The fetch calls in the frontend mapped almost 1:1 to what the FastAPI layer would later do in production. This version also introduced the knowledge mind map: a D3 force graph spanning Web Dev, Data Viz, Infrastructure, AI Tools, Design, Strategy, Security, Finance, and Blockchain. Labels appear progressively as you zoom in — category tags at scale > 0.5, full labels at > 1.1, description snippets at > 3. Three separate SVG text layers managed independently.

Stage 3 — The Grid (Production)Team Intelligence Hub

The full version: real authentication, real persistence, real deployment, real users. System architecture: Browser → HTTPS → Enterprise PaaS (injects x-user-email + x-user-id identity token headers) → FastAPI (Python 3.12, Uvicorn ASGI) → SQLite (WAL mode, seeded from employee JSON on first boot). Why FastAPI: Pydantic models gave automatic request validation and typed contracts. Async lifespan hooks handled DB initialization cleanly. Self-documenting API at no extra cost. Why SQLite with WAL mode: at 100 users with read-heavy patterns, a managed database would have been overengineered. Write-Ahead Logging handled concurrency safely. Choosing not to use a managed database was the right call. Identity resolution: the PaaS injects x-user-email and x-user-id headers on every authenticated request — confirmed by building a debug endpoint and dumping live headers in production. Final implementation checks user identity token first (immutable), falls back to email, falls back to DEV_USER_EMAIL for local dev. Deployment: CI pipeline — container build, signing, push to enterprise registry, deploy via deploy.yaml on every push to main. Two-region (us-west-2 + us-east-1) with traffic-based autoscaling. Multi-stage Dockerfile with non-root user — a PaaS requirement and container security best practice.

The Standing System

The availability traffic light answers one question: what is this person's current bandwidth? Standing answers a different question: has this person been reviewed by their lead recently? Every 90 days, all standing values automatically flip to red. On sign-in, team leads see an alert for their direct reports and must confirm each one, resetting to green with a timestamp. Leads can manually flag someone red at any time with a required written note. The 90-day trigger is a forcing function, not a reminder. Confirmation is required, not optional. The timestamp is stored — the review history is auditable. An earlier version ran two separate cycles (90-day eligibility reset and 60-day standing review). Identifying that redundancy and collapsing it into one coherent system — one mechanism, one cycle, two surfaces — was a design improvement shipped during active development.

Outcome

Production system live on enterprise internal infrastructure. Two-region deployment. 100 users. Full audit and standing confirmation system replacing manual spreadsheet processes. Public portfolio version (thevillanelle.github.io/team-intelligence-hub) accessible from any network with zero external dependencies. Supabase-backed personal version (thevillanelle.github.io/studio) validating the PostgreSQL approach that informed the production design. End-to-end solo: concept, data collection, design, development, and deployment. Extended with a 50-test pytest suite covering RBAC, employee and project CRUD, standing logic, privacy filtering, streaming responses, and performance thresholds. Full production observability: structured JSON logging, request timing middleware, per-route error and 403 tracking, standing confirmation counters, and a live /api/metrics endpoint. Weekly directory sync adds new employees automatically. Real-time monitoring via Datadog across both deployment regions. Extended further through v1.5.9: OWASP security headers on every response; volunteer interest system with structured project roles; SQLite-backed issue tracking with status workflow; avatar photo upload via multipart form; onboarding stepper for new team members; identity-aware Notes (author-stamped, persistent, with custom background color picker); unified traffic-light availability component across all card types; project page redesign with dedicated Teams admin tab and inline field editing throughout.

The Problem

When a team grows from 13 to 100, the spreadsheet that tracked everything stops working. Managers couldn't answer basic questions: who has capacity, who knows what, what's actually being worked on. The data existed — scattered across emails and docs. The system didn't.

What It Does

A platform where every team member has a profile showing their skills, current projects, and availability. Leads get a live view of the whole team at a glance — who's free, who's stretched, what's being worked on. Projects have their own pages with team rosters, status, and objectives — all editable inline by the assigned lead without a separate edit mode.

The Standing Review System

Every 90 days, each team member's review status automatically resets. Leads sign in and see which of their direct reports are due for a conversation — and they can't just click through. Each confirmation requires active acknowledgment and is logged with a timestamp. It's a forcing function, not a reminder. Four real check-ins per year with every person, guaranteed, with a permanent record.

What Makes It Different

Three separate builds to get here — a static prototype to figure out what it needed to do, a proof-of-concept to validate the backend approach, and a full production deployment as the real thing. The process wasn't planned in phases. Each build exposed its own limitations and made the next one possible.

Outcome

A 100-person production platform, built solo from a spreadsheet. The public version is live at thevillanelle.github.io/team-intelligence-hub. The production system is deployed on enterprise infrastructure, serving a real team every day.

Risk Training Academy

Full-stack compliance training platform — knowledge base, decision matrix, scored simulator, geography game, policy acknowledgment system with audit trail, and an AI case work assistant — server-side backend with structured observability, deployed on enterprise infrastructure

The investigations team had no searchable knowledge base, no interactive decision tool, and no training simulator. This built all four — plus a standalone geography training game and a user guide — on a FastAPI backend with a structured KB data layer. 64 KB entries, 40 training scenarios, 135 geography questions, interactive decision matrix for all 4 case types. Deployed on enterprise infrastructure. Works on mobile.

Background

The investigations team works a high-stakes queue: sanctions screening across multiple clients (NovaPay, DevConnect, IdentityCore), four case types, dozens of edge cases, and procedure documents scattered across an internal site with broken deep links and a Cross-Origin-Opener-Policy header that blocks direct external navigation. New investigators had to learn the system by reading dense procedure docs in isolation, with no way to practice before hitting a live queue. The tools existed. The training infrastructure didn't. The gap between "read the procedure" and "work a case confidently" was the problem to solve.

The Codex — Knowledge Base

63 entries across 15 categories: glossary (35), tool (8), best-practice (6), job-aid (3), and domain-specific categories covering procedures, review model, match determination, decisioning, client reference sections. Search is case-insensitive regex across title, body, and tags — handles malformed input and special characters gracefully, whitespace-only returns all 63 entries. The COOP header workaround: every KB card includes a "Copy source URL" button — the Cross-Origin-Opener-Policy header on the internal procedure site blocks direct navigation, so deep links copy to clipboard instead. The KB source of truth lives in kb-data.js (human-editable), which is ingested at deploy time into kb_data.json — the FastAPI backend reads from JSON, not JS.

The Oracle — Decision Matrix

Interactive step-by-step decision tree for all four case types: With PII, IdentityCore Without PII, Companies, and Batch Free Developer. Investigators answer branching questions — entity type, name match, DOB, location — and receive the correct action (Clear, Escalate, or Suspend), the exact reason code, a plain-English explanation, and a full breadcrumb trail of their decision path. A progress bar tracks position through each tree. Built from the official decisioning matrix plus procedural context from all four client procedure docs. Covers edge cases: China company naming, the full street address override, the previously suspended batch_free_developer scenario. The trees are plain JavaScript objects. Traversal is a simple recursive function.

The Trials — Training Simulator

40 scenarios across three difficulty levels. Initiate (13): clear variances, single-factor decisions, false positives. Investigator (13): China naming, 2/3 name match with account age, expired PII, AKA traps, YOB shortcuts. Olympian (14): address match exception, ownership percentages, previously suspended companies, the deceased-date-only escalation, system notes traps, the gibberish flag, the Iran connection override. Fisher-Yates shuffle ensures a different order every session. Wrong-answer pedagogy: every incorrect answer shows full correct reasoning before advancing. Score, 3 lives, rank progression (Trainee through Lead Investigator), streak bonus at 3+ consecutive correct.

Where in the World? — Geography Training Game

135 questions across 4 modes: Script Intel (identify writing systems including 조선 and ایران), Country Decoder (countries from native-language names including all 3 sanctioned countries), Name Origins (Zhang Wei, José, Reza, Nguyễn, Giuseppe), Sanctioned or Safe? (all 3 sanctioned countries, all 5 Ukrainian sanctioned regions, city-level address scenarios). Compliance-critical questions flagged red, worth 150 points. An investigator who can't recognize that ایران is "Iran" in Persian script is more likely to miss a true match. Geography training is compliance training.

Tools Page + Date Corrections

Date conversion utilities for compliance-critical formats: Japanese imperial era (Shōwa, Heisei, Reiwa) and Taiwanese ROC calendar. The Japanese converter corrects an off-by-one error in the official Job Aids page — Shōwa Year 1 = 1926, not 1927. The correction is sourced from JACAR archives, documented, and flagged with a visible warning banner. Nine location-specific visual themes — one per office — each remapping the full CSS custom property set. Theme persists in localStorage.

Backend & Observability

FastAPI backend serving the KB via /api/kb and /api/kb/{id}. Every request emits structured JSON logs. /metrics endpoint exposes live request counts, error rates, and avg response times per route. Multi-stage Dockerfile with non-root user. GitHub Actions CI/CD: seeds kb_data.json from kb-data.js, builds container, signs, pushes to enterprise registry, deploys to production infrastructure.

Outcome

Full training and reference platform live on enterprise production infrastructure: thevillanelle.github.io/risk-training-academy. 64-entry searchable knowledge base. 40 training scenarios across 3 difficulty levels. 135-question geography game. Interactive decision matrix for all 4 case types. Policy acknowledgment system with full timestamp audit trail — every team member signs off on key policies in-platform. Report Error integration on every KB card, routing to the team's issue tracking system. Corrected date tools with documented bug fix. 9 location themes. 18-test pytest suite covering health, KB search, entry lookup, malformed input, schema validation, corrupted KB handling, performance thresholds, and case-insensitivity.

The Problem

New team members had to learn a complex, high-stakes system by reading dense procedure documents in isolation — no practice, no feedback, no way to build confidence before working real cases. The knowledge existed. The training infrastructure didn't.

The Knowledge Base

64 searchable entries covering every concept, tool, and procedure the team works with. Search handles typos and partial matches. Every card that references an external source includes a "copy link" button because the source site's security settings block direct navigation — a real workaround for a real constraint.

The Decision Tool

Step-by-step guidance for any of the four case types the team handles. Enter the details of a case and it walks through every decision point — telling you not just the outcome, but exactly why, with the full decision trail visible at the end.

The Practice Simulator

40 scenarios across three difficulty levels. Every wrong answer shows the full correct reasoning before moving on — the goal isn't to score well, it's to understand why. Rank progression gives you a sense of where you are.

Why the Geography Game

Recognizing that a name or script is from a specific part of the world is a real job skill — and one that's hard to teach with documents. 135 questions covering writing systems, country names in native languages, and specific regional detail. Geography training is compliance training.

Outcome

A full training platform live at thevillanelle.github.io/risk-training-academy — usable from any device, with no login required. Policy acknowledgment system with a permanent audit trail. Built solo, from identifying the gap to shipping the platform.

EQX Doubles Finder

Workout scheduling across 36 NYC Equinox locations — three repos, three platforms, one SQL function throughout

Expand
PostgreSQLPLpgSQLSupabaseJavaScriptPythonFlaskJinja2SwiftSwiftUIXcodeGitHub ActionsRailwayMVVMAsync/AwaitHaversineMTA DataRow Level Security

A personal problem built to production across three complete platform rewrites — three separate repos, three languages, one SQL function that survived all of them. The project wasn't planned in phases. It was iterated into them, each rewrite driven by a concrete limitation hit in the previous one.

Background

Booking back-to-back Equinox classes across New York's network is genuinely complex: which two classes pair well by type, do the times actually work, is the second location walkable or a 40-minute subway ride, which clubs have a pool or steam room. The network spans 36 locations across Manhattan and Brooklyn. No tool existed to answer all of these questions together. The mental overhead was real and recurring, so the tool got built.

Phase 1 — The SPAthevillanelle.github.io/eqx-doubles

Browser → Supabase direct. Schema: clubs table (36 NYC locations with lat/lng), classes table (club_id FK, category, day_of_week, start_mins/end_mins as minutes-since-midnight). Four indexes, Row Level Security, GitHub Actions PDF ingestion every 6 hours. The core logic is find_doubles() — a PLpgSQL stored procedure invoked via db.rpc(). Eight parameters: array of club IDs, day of week, two category values, max gap, pair order (either/1first/2first), time window bounds. Category aliases expand via a CTE. The original schema.sql had a PostgreSQL bug — RETURNS TABLE column names aren't referenceable inside the function body. fix_find_doubles.sql patches it with positional ORDER BY (ORDER BY 5, 15). That fix is committed as its own file. Transit router: buildTransitCache() pre-computes 630 club pairs on boot. Four transit types: same, walk (haversine, 1.3 route factor, 80m/min), quick-train, extended. MTA line badges use official color assignments. pickTop3() geographic profiling: morning→downtown/Brooklyn, afternoon→midtown, evening→uptown. Problem discovered in production: Supabase credentials visible in HTML source.

Phase 2 — The Python Rewriteweb-production-31654.up.railway.app

Browser → Flask → Supabase. Credentials moved to server environment variables. app.py: all routes, CLUBS and CATEGORIES as Python lists of dicts, app entry. supabase_client.py: one module, reads from env with hardcoded fallbacks. requirements.txt: three dependencies (flask, supabase, python-dotenv). Procfile + railway.json for Railway deployment. Club checkboxes and category dropdowns generated server-side via Jinja2 {% for %} loops. All interactive behavior stays JavaScript in the browser, carried over from Phase 1. find_doubles() unchanged — same function, same parameters — now called via the Supabase Python SDK.

Phase 3 — The iOS Appgithub.com/thevillanelle/eqx-doubles-ios

Swift + SwiftUI + Xcode. Same Supabase backend. Same find_doubles() via REST API. Built explicitly to learn compiled languages — the README documents interpreter vs. compiler, what Xcode is, what SwiftUI's declarative model means, and the full compile → binary → simulator → run flow. Models/ (Club, DoubleResult, SearchParams as Codable structs), Services/ (SupabaseService.swift, async/await), Views/ (SearchView, ResultsView, ResultCard, ClubsView, SplashView), SearchViewModel (@Published, MVVM). TransitHelper.swift ports the haversine calculation and subway routing from JavaScript to Swift.

The Through-Line

find_doubles() was written once, in PLpgSQL, in Phase 1. It ran in Phase 2 behind a Flask server. It runs in Phase 3 behind a SwiftUI app. Three separate repos. Three complete rewrites of the presentation and infrastructure layer. Zero rewrites of the core logic. The algorithm lives where the data lives. The application layer is replaceable. Each rewrite had a specific reason: Phase 1 proved the concept and found the credential problem. Phase 2 fixed it and added server-side rendering. Phase 3 was a learning exercise in compiled languages using a problem that was already solved.

The Problem

Booking back-to-back workout classes across New York's 36 Equinox locations is surprisingly hard. The times have to work. The travel between locations has to be realistic. The class types need to pair well. No tool existed to figure all of that out at once — it required mentally juggling schedules, maps, and class categories simultaneously.

What It Does

You choose your preferences — which locations, what class categories, what time window, how much travel you're willing to do — and it finds class pairs that actually work, with transit details included. A stored procedure written once does all of this work regardless of which app is calling it.

Three Complete Rewrites

The project started as a browser app, became a Python web server (to fix a credential security problem the first version had), and then became a native iOS app — as a deliberate exercise in learning compiled languages. Each rewrite had a specific reason. The core matching logic was written once and ran through all three without changes.

The Through-Line

Three separate repos. Three complete rewrites of the presentation layer. Zero rewrites of the core algorithm. The logic lives in the database. The app is replaceable. That's the point.

The Platformritualware.app

The marketing site is a proper multi-page React app — not a landing page. Routes: Home (hero + suite tiles), Platform (six-app overview + one-profile philosophy), per-app marketing pages for each tool, About, and a live Status page. Seed Your Ideas: Compliance Edition

End-to-end idea submission program with a full automation layer — built entirely on tools the team already had open

Expand
Slack WorkflowsQuip AutomationMCP IntegrationAI Agent DevInternal AI PlatformProgram DesignSystems ArchitectureTechnical Writing

Phase 1 built the intake pipeline. Phase 2 closed a feedback loop that Phase 1 didn't solve — identified through direct program evaluation and fixed with a second structural build. Every piece of the system, every document, and every communication asset was designed and produced by one person.

Background

the organization had an investigator idea submission program. Compliance didn't. As the team expanded, there was no structured way for investigators to surface ideas, no place for leadership to track them, and no visibility into what was being considered or actioned. The gap was identified and the case for a compliance-specific program was made.

Phase 1 — Building the Pipeline

The design constraint was adoption: a system that required new tools or new behaviors wouldn't get used. The pipeline was built entirely on Slack and Quip — tools the team already had open every day. A Slack workflow was designed and deployed as the submission interface. Investigators fill out a structured form — name, category, idea description, business impact — without leaving Slack. The workflow populates a purpose-built Quip tracking spreadsheet automatically, giving leadership a centralized, real-time view of all submissions. Built with a program partner. Handed off to a dedicated program lead, responsible for day-to-day management, idea evaluation, and working submissions with the broader leadership team.

The Gap Identified

Quarterly presentations were established as the communication mechanism after handoff. Through those presentations, a structural problem became clear: investigators were submitting ideas and hearing nothing back. No acknowledgment, no status update, no close-out. They had no way to know whether their idea had been seen, considered, actioned, or shelved. The quarterly format was reporting to leadership, not communicating back to investigators. Something in the architecture needed to change.

Phase 2 — Closing the Loop

A new system was designed with one binding constraint: it couldn't add workload to the leadership team already running the program. All trigger points had to map to things leaders were already doing in Quip. Quip structural redesign: clearly designated internal notes column (excluded from all automated outputs), Actions Taken column as single source of truth for close-out communications, format guidelines defined to ensure clean AI agent output. Quip-to-Slack integration: edits to the document — including the Complete checkbox trigger — post automatically to the channel. A leader notification step was also added to the original Slack workflow, triggering an alert on every new submission. Four email templates, designed and copy-edited to organizational voice standards, mapped to the four states an idea can be in: Stage 1 (submission confirmation), Stage 2 (active review notification, triggered when the first Actions Taken entry is logged), Stage 3A (implemented — full close-out with a dated timeline compiled from the Actions Taken history), Stage 3B (tabled/declined — honest, respectful close-out with clear reasoning and an actionable next step). Every trigger point maps to an action a leader was already taking. No new behaviors required.

The Custom AI Agent

A custom agent was built on the org's AI platform and connected to the Quip document via MCP (Model Context Protocol) integration. The agent reads live data directly from the spreadsheet, compiles the Actions Taken history, and produces a formatted, policy-compliant email draft on demand — in the correct organizational voice, with team member names never surfaced to investigators. Any team member can generate the correct email for any stage with a single sentence prompt. The agent's instructions, constraints, tone standards, privacy rules, and operational logic were written from scratch. The goal was a tool any member of the team could pick up and use correctly without knowing how it was built.

Program Asset Suite

Leader program overview (visual explainer of the full pipeline, PDF-ready and email-safe). Investigator guide (plain-language visual guide including what communications to expect at each stage and when). Full email journey reference (all four templates in sequence with trigger annotations). Seed Your Ideas setup guide (step-by-step internal AI onboarding with agent instructions ready to copy and deploy). General AI platform getting started guide (broader onboarding covering GenAI training, developer tooling setup, MCP servers, tokens, and custom agent creation). Leadership elevator pitch. Investigator Slack blast.

Outcome

43+ ideas collected across multiple submission cycles. Over half actioned or completed, with documented outcomes for each. 4 email templates designed, copy-edited, and ready to deploy against the existing backlog. 1 custom AI agent live and operational — close-out email drafting reduced to a single prompt. 6 program documents covering leadership, investigators, and team onboarding. Zero additional tooling required — the full system runs on Slack, Quip, and the org's AI platform. Zero ongoing maintenance burden — the agent handles drafting, Quip-to-Slack handles alerting. Any team member can run the program end-to-end using the shared documentation and agent.
